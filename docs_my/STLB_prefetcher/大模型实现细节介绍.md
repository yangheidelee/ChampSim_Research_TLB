# STLB Prefetcher Infra 实现详解

> 本文档以当前工作树中的实际代码为准，聚焦为 STLB-local translation prefetcher 新增的 ChampSim infrastructure，而不是某个具体预取算法。

## 1. 最终语义概览

当前实现已经把 STLB prefetcher 从普通 data-cache line prefetcher 中分离出来，形成一套独立的 translation-prefetch 接口，但仍复用 ChampSim 原有的模块发现、JSON 配置、`CACHE` tag/MSHR/fill pipeline 和 channel backpressure 语义。

整体链路是：

```text
real data translation request
        |
        v
      STLB lookup
        |
        +-- hit --------------------------> stlb_prefetcher_operate(hit=true)
        |
        +-- miss -- handle_miss succeeds -> stlb_prefetcher_operate(hit=false)
                                               |
                                               v
                                      predictor trains/generates VPN
                                               |
                                               v
                                      prefetch_translation(VA, metadata)
                                               |
                                  accepted into STLB internal_PQ
                                               |
                                  low-priority STLB tag pipeline
                                               |
                         +---------------------+--------------------+
                         |                                          |
                  merge existing STLB MSHR                 allocate STLB MSHR
                                                                    |
                                               prefetch_as_load=false
                                                                    |
                                                                    v
                                                          STLB -> PTW PQ
                                                                    |
                                                PTW serves RQ before PQ
                                                                    |
                                                       page-table walk/fill
                                                                    |
                                                                    v
                                                        STLB block + stats
```

实现的核心约束是：

- STLB demand translation 和 vBerti Permit-PGC translation 仍使用 PTW RQ；
- STLB-local prefetch 使用 PTW PQ；
- PTW RQ 严格优先于 PQ，两者共享 `MAX_READ` 带宽；
- STLB 本地 PQ 满时才直接拒绝本次提交；
- 已经进入 pipeline 的请求在 STLB MSHR 或 PTW PQ 暂时不可用时保留并重试；
- STLB-local 预取的 origin、ownership 和统计与 vBerti/data-prefetch 独立；
- 不配置 STLB prefetcher 时仍是 `no`，旧 JSON 的默认语义不变。

## 3. STLB 专用模块接口

### 3.1 Predictor 继承的基类

专用基类位于 [`inc/modules.h`](/home/zcq/git_prj/ChampSim/inc/modules.h:189)：

```cpp
struct stlb_prefetcher : public bound_to<CACHE> {
  explicit stlb_prefetcher(CACHE* cache);

  bool prefetch_translation(
      champsim::address virtual_address,
      uint32_t prefetch_metadata = 0) const;
};
```

具体模块应当继承 `champsim::modules::stlb_prefetcher`，而不是普通的 `champsim::modules::prefetcher`。例如 [`sp.h`](/home/zcq/git_prj/ChampSim/prefetcher_stlb/sp/sp.h:9)。

`prefetch_translation()` 是 translation-prefetch 对应的发射接口；当前 STLB predictor 不使用 `prefetch_line()`。

### 3.2 五个可选回调

| STLB callback | 对应的 cache-prefetcher 生命周期 | 当前语义 |
|---|---|---|
| `stlb_prefetcher_initialize()` | `prefetcher_initialize()` | 模拟开始前调用一次，初始化 predictor state |
| `stlb_prefetcher_operate(const stlb_prefetcher_context&)` | `prefetcher_cache_operate(...)` | 接收经过过滤的 STLB demand hit/已成功消费的 miss |
| `stlb_prefetcher_fill(const stlb_prefetcher_fill_context&)` | `prefetcher_cache_fill(...)` | STLB translation 被填入前接收 fill 和 victim 信息 |
| `stlb_prefetcher_cycle_operate()` | `prefetcher_cycle_operate()` | 每个 STLB cycle 调用，用于后台状态机或延迟发射 |
| `stlb_prefetcher_final_stats()` | `prefetcher_final_stats()` | 模拟结束时打印 predictor-private 统计 |

精确的编译期 callback detection 定义在 [`inc/modules.h`](/home/zcq/git_prj/ChampSim/inc/modules.h:196)。它们是可选函数：predictor 可以只实现需要的生命周期。

当前六种基础 miss-stream predictor 通常只需要 initialize、operate 和 final-stats；不实现 fill/cycle 不代表 infra 缺少这两个信号。

### 3.3 Operate context

[`stlb_prefetcher_context`](/home/zcq/git_prj/ChampSim/inc/modules.h:157) 为：

| 字段 | 语义 |
|---|---|
| `v_address` | 当前 translation access 的虚拟地址 |
| `vpn` | `v_address >> LOG2_PAGE_SIZE` |
| `ip` | 触发该 translation access 的 instruction PC |
| `instr_id` | ChampSim instruction identifier |
| `cpu` | 触发 CPU |
| `asid[2]` | ChampSim 现有的两字节 ASID 对；infra 不解释为两种固定含义 |
| `type` | `LOAD/RFO/PREFETCH/...` access type |
| `origin` | translation 的真实来源 |
| `hit` | 本次 STLB lookup 是否 hit |
| `useful_prefetch` | 当前 `try_hit()` 计算的通用 cache-prefetch useful 标志，见下文限制 |
| `warmup` | 当前 phase 是否为 warmup |
| `metadata` | 随请求传到当前 lookup 的 32-bit metadata |

这些字段由 [`operate_stlb_prefetcher()`](/home/zcq/git_prj/ChampSim/src/cache.cc:903) 从真实 `tag_lookup_type` 中构造，不是 predictor 自己推测出来的。

### 3.4 Fill context

[`stlb_prefetcher_fill_context`](/home/zcq/git_prj/ChampSim/inc/modules.h:172) 另外提供：

| 字段 | 语义 |
|---|---|
| `v_address/vpn` | 将要 fill 的 translation |
| `evicted_v_address/evicted_vpn` | 被选中 victim 的虚拟地址/VPN |
| `set/way` | fill 目标位置 |
| `type/origin` | fill MSHR 当前的 access type 和 translation origin |
| `prefetch` | 当前实现中等于 `fill_mshr.type == PREFETCH` |
| `cpu/asid/instr_id/warmup` | fill 对应的上下文 |
| `metadata` | 从 data promise 返回的 32-bit prefetch metadata |

fill wrapper 在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:930) 构造这个 context。调用点在写入新 block 之前，因此 predictor 能看到真实 victim，见 [`handle_fill()`](/home/zcq/git_prj/ChampSim/src/cache.cc:546)。

需要注意：STLB-local prefetch MSHR 如果后来合并了 demand，ChampSim merge 后的有效 `type` 可以变成 `LOAD`，但 ownership 仍是 STLB prefetch。因此当前 fill context 的 `prefetch` 不能被理解为“这次 fill 一定由 STLB-local prefetch 拥有”；中央统计使用独立的 `stlb_prefetch_tracked` 判断 ownership。

### 3.5 Metadata 能力边界

当前接口可以携带“复杂但已编码”的 32-bit metadata：

```text
operate context.metadata
    -> prefetch_translation(target, metadata)
    -> request/tag lookup/MSHR/PTW response
    -> fill context.metadata
```

它不是任意 C++ 对象或无限长 context。如果算法需要大型状态，推荐在 predictor 内部保存表，只用 metadata 携带 index/ID/bit-field。

## 4. Module loader 和 type-erasure 如何接入新接口

ChampSim 仍使用原有 `CACHE::prefetcher_module_concept/model` 加载模块，没有平行再造一套 JSON loader。

### 4.1 增加的虚函数边界

[`prefetcher_module_concept`](/home/zcq/git_prj/ChampSim/inc/cache.h:392) 新增：

```cpp
virtual void impl_stlb_prefetcher_operate(
    const champsim::modules::stlb_prefetcher_context&) = 0;

virtual void impl_stlb_prefetcher_fill(
    const champsim::modules::stlb_prefetcher_fill_context&) = 0;
```

template model 在 [`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:585) 中对 tuple 内的每个 predictor 做编译期检测；只有定义了正确 STLB 函数名和签名的模块才会被调用。

### 4.2 两套函数名不是运行时模式开关

initialize、cycle 和 final-stats wrapper 会分别检查普通和 STLB 函数名，见 [`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:529) 与 [`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:609)。

这表示：

- 模块只实现 `stlb_prefetcher_initialize()`，则只调它；
- 模块只实现 `prefetcher_initialize()`，则只调普通版；
- 同一个类同时实现两个名字，则两个都会调用。

对 operate/fill 而言，`CACHE` 调用点还做了硬隔离：

- STLB lookup 不调普通 `prefetcher_cache_operate()`，见 [`try_hit()`](/home/zcq/git_prj/ChampSim/src/cache.cc:640)；
- STLB fill 调专用 fill，非 STLB data fill 才调普通 fill，见 [`handle_fill()`](/home/zcq/git_prj/ChampSim/src/cache.cc:585)。

因此 STLB 算法应统一使用 `stlb_*` 回调和 `prefetch_translation()`。从 C++ 类结构上同时写两套函数是可以的，但不是本实现推荐的 STLB 模块写法。

## 5. Predictor 何时收到 operate 回调

### 5.1 中央过滤条件

[`operate_stlb_prefetcher()`](/home/zcq/git_prj/ChampSim/src/cache.cc:903) 要求：

```cpp
is_stlb()
&& is_data_demand_origin(handle_pkt.translation_source)
&& !module_is_instr(handle_pkt)
&& should_activate_prefetcher(handle_pkt)
```

实际语义是：

- 只有 STLB 会调 STLB predictor；
- 只接受真实 data-demand translation origin；
- instruction translation 不训练；
- access type 必须包含在 JSON `prefetch_activate` 中；
- predictor 自己的 `prefetch_from_this` 请求不会再触发 predictor。

因此当前 STLB-local 请求、vBerti translation request、instruction translation 和 PTW 内部访问都不会训练这六种 predictor。

普通 data-cache prefetcher 的中央层只排除 instruction request，并根据 `prefetch_activate` 过滤，它不使用这个 `DEMAND_DATA` translation-origin 过滤。这是 STLB 专用接口为了明确 miss-stream 语义增加的边界。

### 5.2 Hit 调用点

`try_hit()` 先完成 tag comparison；如果是 STLB hit，立即调用专用 operate，见 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:622) 和 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:645)。

六种基础 predictor 在自己的 operate 中执行：

```cpp
if (context.hit)
  return;
```

所以当前算法是 miss-trained，但 infra 仍提供 hit event，以便以后实现 access-trained 算法。

### 5.3 Miss 调用点与 stall retry

miss 不是一检测到就通知 predictor。关键逻辑在 [`CACHE::operate()`](/home/zcq/git_prj/ChampSim/src/cache.cc:1099)：

```cpp
const bool handled = handle_miss(pkt);
if (handled)
  operate_stlb_prefetcher(pkt, false, false);
return handled;
```

`handle_miss()` 成功包括两种情况：

1. 请求合并到已有 STLB MSHR；
2. 下层队列接收请求，并在需要响应时成功分配新 STLB MSHR。

如果 STLB MSHR 满或下层 RQ/PQ 暂时无法接收，`handle_miss()` 返回 `false`，请求保留在 `inflight_tag_check` 中下周期重试，本周期不通知 predictor。

所以当前 STLB 语义是：

- 同一个 request 的 stall retry 不重复训练；
- 请求成功消费后只训练一次；
- 之后另一条真实 load 访问同一 VPN，即使它也合并到同一 MSHR，仍是新的 access，会再训练一次。

这与普通 cache prefetcher 完全相同吗？不完全相同。普通 cache callback 在 [`try_hit()`](/home/zcq/git_prj/ChampSim/src/cache.cc:640) 中、在 miss 资源是否能接收之前调用，因此可能在 stall retry 上重复观察同一 access。STLB 专用 miss callback 故意选择成功消费后再训练，避免为 DP/H2P/ASP/MASP 伪造 0 distance/stride。

### 5.4 触发 demand 与它生成的 prefetch 的先后顺序

由于 miss callback 发生在 `handle_miss()` 成功之后，触发这次训练的 demand 已经先合并/分配 STLB MSHR 并进入 PTW RQ，然后 predictor 才能把新候选项放入 STLB 本地 PQ。这个顺序保证了触发 demand 本身不会被同次 callback 产生的 prefetch 抢在前面。

## 6. `prefetch_translation()` 的精确语义

predictor-side wrapper 在 [`src/modules.cc`](/home/zcq/git_prj/ChampSim/src/modules.cc:26) 中只做一层转发，真正的提交逻辑在 [`CACHE::prefetch_translation()`](/home/zcq/git_prj/ChampSim/src/cache.cc:1274)。

### 6.1 提交顺序

1. 如果绑定的 cache 不是 STLB，直接返回 `false`，不增加 STLB requested。
2. 增加 `stlb_prefetch_requested`。
3. 检查 `internal_PQ.size() >= PQ_SIZE`。
4. 队列满时返回 `false`。
5. 否则构造一个 STLB-local translation request 并放入 `internal_PQ`。
6. 增加 `stlb_prefetch_issued`，返回 `true`。

包的关键字段是：

```text
type                  = PREFETCH
translation_source    = STLB_PREFETCH
address/v_address     = predictor 提交的 virtual address
is_translated         = true
response_requested    = true
stlb_prefetch_tracked = true
prefetch_from_this    = true   // internal_PQ -> tag packet 时设置
cpu/asid              = 当前 predictor callback 上下文
pf_metadata           = predictor 传入的 32-bit metadata
```

`is_translated=true` 表示它已经是直接要在 STLB 查找的目标虚拟页 translation，不再把该候选项当作一条还需先经 DTLB 翻译的 data-cache line request。STLB miss 后它才会下发 PTW。

### 6.2 `bool` 只表示 STLB 本地 PQ 是否接收

| 返回值 | 精确含义 |
|---|---|
| `true` | 候选项已进入 STLB `internal_PQ` |
| `false` | 未进入 STLB `internal_PQ`；通常是该队列满，或函数被错误用于非 STLB |

这个返回值不表示：

- 最终 STLB hit 还是 miss；
- 是否合并 MSHR；
- 是否已进 PTW PQ；
- 最终是否 fill/useful/late/useless；
- 具体是什么下游拒绝原因。

predictor 观察到有效 miss 后对自身 history/table 的更新不依赖这个返回值；提交失败不回滚训练。

### 6.3 隐式 CPU/ASID 上下文

operate/fill wrapper 在调用 predictor 之前把当前 request 的 `cpu` 和 `asid[2]` 保存到 `CACHE`，见 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:909) 和 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:935)。`prefetch_translation()` 再使用这个隐式上下文。

当前六种 predictor 都在 `stlb_prefetcher_operate()` 内同步发射，所以上下文与触发 access 一致。

当前边界是：如果以后在 `stlb_prefetcher_cycle_operate()` 中延迟发射，发射接口没有显式 CPU/ASID 参数，将使用最近一次 operate/fill 留下的上下文。当前单核、单地址空间 trace 范围内这不影响结果；若扩展到 multicore/multi-ASID 延迟发射，应将 CPU/ASID 改为 `prefetch_translation()` 的显式参数，或让 predictor 自己保存完整 pending context。

## 7. STLB 本地 PQ 和 tag pipeline

`internal_PQ` 是 `CACHE` 已经存在的本地 prefetch queue，容量由 STLB JSON 的 `pq_size` 决定。STLB prefetcher 没有新造一个脱离 cache pipeline 的无限队列。

### 7.1 Tag-initiation 优先级

[`CACHE::operate()`](/home/zcq/git_prj/ChampSim/src/cache.cc:1050) 的 tag initiation 顺序是：

1. 先处理需要 translation 完成的 stash；
2. 再处理所有 upper-level channel 的 `WQ -> RQ -> PQ`；
3. 最后用剩余 `MAX_TAG` bandwidth 从本 cache 的 `internal_PQ` 取请求。

对应代码在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:1072) 和 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:1084)。

因此 STLB-local prefetch 不仅在 PTW 端是低优先级，在 STLB 本级进入 tag pipeline 时也使用 upper-level 真实请求之后的剩余带宽。

### 7.2 Backpressure 和重试

```text
还未进入 internal_PQ：
    internal_PQ full -> prefetch_translation() == false

已进入 internal_PQ：
    等待 STLB tag bandwidth
    -> 等待 STLB MSHR/合并
    -> 等待 PTW PQ 可接收
    -> 成功后移除
```

已进入 pipeline 的请求不会因为下游一时不可用就被记为 drop。它会占用队列/tag-check 位置并继续重试，这正是模型化 backpressure 的地方。

`pq_drop_rate` 仅描述 predictor 提交到 STLB `internal_PQ` 这一层的 `requested-issued`，不是 PTW PQ 的后续 drop rate。

## 8. STLB MSHR merge 和 ownership

### 8.1 先 merge，再检查 MSHR 是否满

[`handle_miss()`](/home/zcq/git_prj/ChampSim/src/cache.cc:790) 先查找已有 MSHR。如果 VPN 已经有 MSHR，即使当前 MSHR 数量达到容量也可以 merge；只有需要新建 MSHR 时才会在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:827) 因为 `MSHR_SIZE` 停顿。

### 8.2 Ownership 由第一个分配 MSHR 的请求决定

merge 逻辑在 [`CACHE::mshr_type::merge()`](/home/zcq/git_prj/ChampSim/src/cache.cc:381)：

```cpp
retval.stlb_prefetch_tracked = predecessor.stlb_prefetch_tracked;
retval.stlb_prefetch_used = predecessor.stlb_prefetch_used;
```

`predecessor` 是已存在、先分配的 MSHR。因此：

| 先到 | 后到 | 结果 |
|---|---|---|
| demand/vBerti | STLB-local prefetch | prefetch 合并到已有 MSHR，记 `prefetch_mshr_merge`，但不抢走 fill ownership |
| STLB-local prefetch | demand | demand 合并到 prefetch-owned MSHR，立即记 useful + late，fill 仍属于该 prefetch |
| STLB-local prefetch | STLB-local prefetch | 后者记 MSHR merge，ownership 仍属于先到请求 |

这一规则避免“一个 late prefetch 只是 merge 到 demand MSHR，却把 demand 本来会产生的 fill 冒领为自己的 fill”。

### 8.3 后续多个同 VPN demand

如果 prefetch 先建立 MSHR，第一个后来 demand 会将 `stlb_prefetch_used` 设为 true，并只记一次 useful/late，见 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:813)。再后续的同 VPN demand 仍会 merge 到这个已有 MSHR，其 dependency/return target 会记录在 MSHR 中，但因 `stlb_prefetch_used` 已经为 true，不会重复增加 useful/late。

所以一个 late translation prefetch 确实可能成为后面多个同 VPN load 共享的 in-flight walk。这些 load 要等同一个 PTW 结果，但不是每个 load 新建一个 walk；它们的 dependency 会合并到 MSHR。

## 9. STLB 到 PTW 的 RQ/PQ 路由

新建 STLB miss 需要向 PTW 发送时，[`handle_miss()`](/home/zcq/git_prj/ChampSim/src/cache.cc:842) 使用：

```cpp
const bool send_to_rq =
    prefetch_as_load || handle_pkt.type != access_type::PREFETCH;
```

在约定配置 `STLB.prefetch_as_load=false` 下：

| 请求 | `type` | `origin` | STLB -> PTW channel |
|---|---|---|---|
| real data demand | `LOAD` | `DEMAND_DATA` | RQ |
| instruction demand | `LOAD` | `DEMAND_INSTRUCTION` | RQ |
| vBerti/Permit-PGC translation | 仍是 demand-like `LOAD` | `L1D_PREFETCH_*` | RQ |
| STLB-local prefetch | `PREFETCH` | `STLB_PREFETCH` | PQ |

如果存在匹配 MSHR，请求直接 merge，不再向 PTW 的 RQ/PQ 插入新 entry。

如果 PTW PQ 满，`lower_level->add_pq()` 失败，`handle_miss()` 返回 `false`；这个 STLB-local request 保留在 STLB tag-check pipeline 中，后续重试。它不会再次进入 predictor 提交阶段，也不增加 `requested-issued` drop。

## 10. PTW 仲裁、带宽和 MSHR 约束

### 10.1 RQ 严格优先于 PQ

[`PageTableWalker::operate()`](/home/zcq/git_prj/ChampSim/src/ptw.cc:186) 每周期只创建一个：

```cpp
champsim::bandwidth tag_bw{MAX_READ};
```

然后：

1. 遍历所有 upper-level RQ，消耗 `tag_bw`；
2. RQ 都被考虑之后，才遍历 PQ；
3. PQ 只能使用剩余 `tag_bw`。

对应代码在 [`src/ptw.cc`](/home/zcq/git_prj/ChampSim/src/ptw.cc:187) 和 [`src/ptw.cc`](/home/zcq/git_prj/ChampSim/src/ptw.cc:199)。如果 `handle_read()` 本周期不能启动 walk，该 entry 保留在 RQ/PQ 里等待重试。

这里的“严格优先”是服务顺序，不是说 PQ 永远不能服务：当本周期 RQ 没有耗尽 `MAX_READ` 时，PQ 可使用剩余额度。

### 10.2 `max_read` 和 `max_write` 都有效

JSON 生成器将 PTW：

- `max_read` 映射为 `tag_bandwidth`/`MAX_READ`；
- `max_write` 映射为 `fill_bandwidth`/`MAX_FILL`。

映射见 [`config/instantiation_file.py`](/home/zcq/git_prj/ChampSim/config/instantiation_file.py:91)，兼容键名见 [`config/parse.py`](/home/zcq/git_prj/ChampSim/config/parse.py:26)。PTW 在 [`src/ptw.cc`](/home/zcq/git_prj/ChampSim/src/ptw.cc:153) 用 `MAX_FILL` 限制 fill/next-step completion，在 [`src/ptw.cc`](/home/zcq/git_prj/ChampSim/src/ptw.cc:186) 用 `MAX_READ` 限制新读取步骤。

因此 `max_read=2, max_write=2` 不是无效配置。

### 10.3 PTW `MSHR_SIZE` 当前并未真正限流

PTW builder 和 object 中存在 `MSHR_SIZE`，但当前 [`PageTableWalker::operate()`](/home/zcq/git_prj/ChampSim/src/ptw.cc:186) 在从 RQ/PQ 生成 `next_steps` 并插入 PTW `MSHR` 时，没有用 `MSHR_SIZE` 判断是否已满，见 [`src/ptw.cc`](/home/zcq/git_prj/ChampSim/src/ptw.cc:213)。

所以不应将“STLB MSHR=16”等价表述为“PTW MSHR=16”。当前实际起约束作用的是：

- STLB MSHR 容量；
- STLB internal PQ/tag bandwidth；
- STLB -> PTW RQ/PQ 容量；
- PTW `MAX_READ/MAX_FILL`；
- 页表 walk 下层 cache/DRAM 的队列和时序。

如果将来要建模真正的“PTW 同时在途 walk 上限”，需要在 PTW 接收 RQ/PQ 或将 `next_steps` 插入 MSHR 前显式检查 `MSHR_SIZE`。

## 11. Origin 与端到端 provenance

### 11.1 独立 origin

[`translation_origin`](/home/zcq/git_prj/ChampSim/inc/access_type.h:36) 新增：

```cpp
STLB_PREFETCH
```

这使 STLB-local prefetch 不需要冒充 `DEMAND_DATA` 或 `L1D_PREFETCH_*`，也不会被当作 vBerti 请求进入原有统计。

### 11.2 Ownership 传播路径

STLB-local ownership 使用独立的两个布尔字段：

```text
stlb_prefetch_tracked
stlb_prefetch_used
```

它们传播于：

```text
channel::request
    -> CACHE::tag_lookup_type
    -> CACHE::mshr_type
    -> channel::request/response
    -> CACHE::cache_block
```

关键代码位置：

- request/response：[`inc/channel.h`](/home/zcq/git_prj/ChampSim/inc/channel.h:90)；
- tag/MSHR 字段：[`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:103) 和 [`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:145)；
- request -> tag -> MSHR：[`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:349)；
- MSHR -> resident block：[`fill_block()`](/home/zcq/git_prj/ChampSim/src/cache.cc:433)；
- resident block：[`inc/block.h`](/home/zcq/git_prj/ChampSim/inc/block.h:49)。

这些字段是 stats-only provenance：不参与 channel 地址匹配、STLB tag 匹配、replacement victim 选择或 PTW 功能正确性。

PTW 不需要把“谁拥有这次 fill”作为 walk 算法状态：STLB MSHR 在等待 PTW 响应期间保留 ownership，PTW 返回 translation data 后再由 STLB 完成 fill 和统计。

## 12. Fill、useful、late 和 useless

### 12.1 Fill

当 STLB MSHR 真正写入有效 way，且 `fill_mshr.stlb_prefetch_tracked=true` 时，增加 `stlb_prefetch_fill`，见 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:596)。

一个 STLB-local 请求如果只是 merge 到 demand/vBerti 先分配的 MSHR，它不拥有 fill，因此不增加 STLB-local `prefetch_fill`。

### 12.2 Timely useful

当 real demand 命中一个 `stlb_prefetch=true` 且尚未使用的 resident translation 时：

- 将 block 的 `stlb_prefetch_used` 设为 true；
- `prefetch_useful++`；
- 不增加 `prefetch_late`。

代码在 [`try_hit()`](/home/zcq/git_prj/ChampSim/src/cache.cc:678)。

### 12.3 Late useful

当 real demand miss 合并到 STLB-local prefetch 先分配、尚未 fill 的 MSHR 时：

- MSHR `stlb_prefetch_used=true`；
- `prefetch_useful++`；
- `prefetch_late++`。

代码在 [`handle_miss()`](/home/zcq/git_prj/ChampSim/src/cache.cc:815)。后续更多 demand 可以合并到同一 MSHR，但不重复计 useful/late。

### 12.4 Useless

`prefetch_useless` 只在一个已填入、仍未被 demand 使用的 STLB-prefetched translation 真正被替换出去时增加，见 [`handle_fill()`](/home/zcq/git_prj/ChampSim/src/cache.cc:571)。

ROI 结束时：

- 仍驻留在 STLB 的未使用 prefetch；
- 仍在 internal PQ/tag-check/translation stash/MSHR 中的 prefetch；

都不会被强制结算为 useless。它们是未完全观测的 right-censored event。因此：

```text
useful + useless == fill
```

不是当前统计的普遍不变式。

### 12.5 两个 context 字段的当前限制

1. `operate_context.useful_prefetch` 当前来自普通 `way->prefetch` 标志，见 [`try_hit()`](/home/zcq/git_prj/ChampSim/src/cache.cc:631)；STLB-local block 使用独立 `way->stlb_prefetch`。因此该 context 字段不应用来代替中央 `stlb_prefetch_useful` 判断。
2. `fill_context.prefetch` 当前表示 fill MSHR 的有效 `type` 是否为 `PREFETCH`，不一定等于 STLB-local ownership；特别是 demand 合并到 prefetch-owned MSHR 后，`type` 可能变为 `LOAD`。

当前六种基础 predictor 不依赖这两个字段，所以不影响已实现算法。如果新算法需要从 callback 精确区分“首次命中 STLB-local prefetch”或“prefetch-owned fill”，应在专用 context 中增加显式 STLB-local ownership/usefulness 字段，不要推断现有两个字段。

## 13. Too-early 和 pollution 的 shadow-buffer 测量

### 13.1 独立测量结构

STLB-local 统计使用自己的：

```text
stlb_prefetch_too_early_fifo
stlb_prefetch_too_early_shadow
stlb_prefetch_pollution_fifo
stlb_prefetch_pollution_shadow
```

定义在 [`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:308)，不复用普通 data prefetch 或 vBerti cross-page 的 shadow state。

shadow key 是：

```text
(cpu, vpn, asid[0], asid[1])
```

定义在 [`inc/cache.h`](/home/zcq/git_prj/ChampSim/inc/cache.h:163)。容量上限为 `NUM_SET * NUM_WAY`，见 [`too_early_shadow_size()`](/home/zcq/git_prj/ChampSim/src/cache.cc:1672)。它是有限测量结构；候选记录如果在被确认前因容量移出，对应事件不会被计数。

### 13.2 Too early

事件链：

```text
unused STLB-prefetched block is evicted
    -> remember (cpu, vpn, asid) in too-early shadow
    -> later real demand accesses the same translation
    -> demand misses STLB
    -> consume shadow entry
    -> stlb_prefetch_too_early++
```

记录 victim 和 useless 在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:571)，后续 demand miss 消费记录在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:882)，shadow helper 在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:1786)。

too early 不等于所有 useless。它要求后续真实 demand 又访问这个已被提前驱逐的 VPN。

### 13.3 Pollution

事件链：

```text
STLB-local prefetch fill evicts victim V
    -> remember V and whether V was demand-origin
    -> later real demand accesses V
    -> demand misses STLB
    -> stlb_prefetch_pollution_evict++
    -> if V was demand-origin: stlb_prefetch_pollution_demand++
```

prefetch fill 替换 victim 时建立候选记录，见 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:562)；demand miss 消费在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:885)，helper 在 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:1805)。

too-early/pollution shadow 只用于事后归因，不参与 hit/miss、replacement 或请求调度。

## 14. 中央统计的完整定义

### 14.1 Raw counters

[`cache_stats`](/home/zcq/git_prj/ChampSim/inc/cache_stats.h:25) 定义了：

| 计数器 | 增加时机 |
|---|---|
| `requested` | STLB 上调用一次 `prefetch_translation()` |
| `issued` | 成功插入 STLB `internal_PQ` |
| `lookups` | STLB-local request 最终被处理为一次 STLB hit 或成功消费的 miss |
| `hit` | STLB-local request 命中 resident translation |
| `miss` | STLB-local request 作为 STLB miss 成功 merge/下发 |
| `mshr_merge` | STLB-local request 命中已有 STLB MSHR |
| `fill` | STLB-prefetch-owned MSHR 真正填入 STLB |
| `useful` | demand 首次使用 resident 或 in-flight STLB prefetch |
| `useless` | 未使用 STLB-prefetched block 被真正驱逐 |
| `late` | demand 首次合并到 prefetch-owned in-flight MSHR |
| `too_early` | 未使用 prefetch 被驱逐后，demand 又对其 miss |
| `pollution_evict` | prefetch 替换的 victim 后续被 demand miss 访问 |
| `pollution_demand` | 上述 victim 本身是 demand-origin translation |

raw counter 在 hit/miss/fill/eviction 实际事件点更新，不依赖 predictor-private 统计。

### 14.2 文本 log 的派生指标

[`plain_printer.cc`](/home/zcq/git_prj/ChampSim/src/plain_printer.cc:338) 输出最初要求的完整列表。令：

```text
dropped      = max(requested - issued, 0)
timely       = max(useful - late, 0)
demand_miss  = STLB origin-miss 中真实 demand origin 的数量
instructions = ROI retired instructions
```

则：

| 日志指标 | 当前精确公式 |
|---|---|
| `pq_drop_rate` | `dropped / requested` |
| `prefetch_miss_rate` | `prefetch_miss / prefetch_lookups` |
| `prefetch_mpki` | `prefetch_lookups * 1000 / instructions` |
| `prefetch_too_early_among_fill` | `too_early / fill` |
| `prefetch_too_early_among_useless` | `too_early / useless` |
| `prefetch_pollution_among_prefetch_fill` | `pollution_evict / fill` |
| `prefetch_accuracy` | `useful / issued` |
| `prefetch_coverage` | `useful / (useful + demand_miss)` |
| `timely_coverage` | `timely / (useful + demand_miss)` |
| `timely_accuracy` | `timely / issued` |
| `fill_accuracy` | `(timely + late) / fill` |

所有分母为 0 的比率返回 0，不产生 NaN/Inf。

专用的 `pq_drop_rate/prefetch_lookups/prefetch_mshr_merge/prefetch_mpki/timely_coverage/timely_accuracy/fill_accuracy` 行只在 ROI 中 `stlb_prefetch_requested > 0` 时打印。也就是说，真正启用并在 ROI 内调用过 STLB-local predictor 时，普通 `.log` 才显示这组完整专用行；no-mode 不会凭空增加它们。

由于 `timely = useful-late`，在计数器内部关系正常时：

```text
fill_accuracy = useful / fill
```

但实现仍按需求显式打印 `(timely + late) / fill`。由于 ROI 结束右删失，一个 late useful 事件可能已发生而 fill 仍在途，因此短 phase 中这些比率要结合边界理解。

当 ROI 中 `stlb_prefetch_requested > 0` 时，原有 STLB `PREFETCH_access/hit/miss/miss_rate` 标签会改为展示 STLB-local lookups/hit/miss，见 [`src/plain_printer.cc`](/home/zcq/git_prj/ChampSim/src/plain_printer.cc:323)。没有启用 STLB-local predictor 时，原有 aggregate 标签保持旧语义。

### 14.3 Plain log 与 JSON 的区别

- 普通 `.log` 输出 raw counter 和上表的所有派生比率；
- JSON printer 当前输出全部 STLB-local raw counter，见 [`src/json_printer.cc`](/home/zcq/git_prj/ChampSim/src/json_printer.cc:57)；
- JSON 中当前没有另外存储上述所有派生比率，可以从 raw counter 重算。

## 15. Warmup 与 ROI 边界

### 15.1 生命周期时序

ChampSim 在进入所有 phase 之前调用一次 `initialize()`，见 [`src/champsim.cc`](/home/zcq/git_prj/ChampSim/src/champsim.cc:300)。`CACHE::initialize()` 再调 prefetcher initialize wrapper，见 [`src/cache.cc`](/home/zcq/git_prj/ChampSim/src/cache.cc:2152)。

每个 phase 开始时，ChampSim 设置 `operable.warmup` 并调 `begin_phase()`，见 [`src/champsim.cc`](/home/zcq/git_prj/ChampSim/src/champsim.cc:157)。

因此：

- `stlb_prefetcher_initialize()` 在 warmup 前只调一次；
- warmup -> ROI 边界不会重新 initialize predictor；
- predictor table/history/confidence 保留到 ROI；
- warmup 期间 operate/fill/cycle 正常调用，prefetch 也正常发射并影响微架构状态。

`warmup` 被显式放入 STLB context，是为了让 predictor 自己决定私有统计是否计入 ROI。当前六种 predictor 在 warmup 照常训练/发射，但不增加自己的 ROI-only private counters。普通 cache prefetcher 的 context 没有这个显式参数，这是两套接口在模块侧可见信号上的差异，不是说 STLB warmup 期间停止训练或停止发射。

### 15.2 ROI 开始时清理什么

[`CACHE::begin_phase()`](/home/zcq/git_prj/ChampSim/src/cache.cc:2158) 在 STLB 上：

- 清零新 phase 的中央 `sim_stats/roi_stats`；
- 清空 STLB-local too-early/pollution shadow 结构；
- 清除 resident block 上的 STLB-local ownership/used 归因标志；
- 清除现有 STLB MSHR、internal PQ、inflight tag-check 和 translation stash entry 上的 STLB-local provenance 标志。

但它不会：

- invalidate 已 warm 好的 STLB translation；
- 清空真实 MSHR/queue 请求本身；
- 清空 predictor 内部状态。

结果是：warmup prefetch 可以通过已驻留 translation 或仍在途请求影响 ROI 性能，但因 provenance 在 ROI 开始被切断，不归因到 ROI 内的 STLB-local useful/late/useless/too-early/pollution。这是“保留 warmup 微架构状态，但 ROI 仅评价 ROI cohort”的边界。

与普通 cache prefetcher 相比，当前 STLB-local 额外清除了自己的 block/MSHR/queue provenance；普通 `way.prefetch` 并没有在这一段代码中以同样方式清除。这是 STLB-local 为 ROI-only ownership 归因作的有意设计。

### 15.3 ROI 结束

[`CACHE::end_phase()`](/home/zcq/git_prj/ChampSim/src/cache.cc:2223) 只把 phase 中已发生的 STLB-local counter 复制到 `roi_stats`。它不扫描驻留 block 或 in-flight request 并强制将它们标记为 useless。

`finalize_tlb_cross_prefetch_stats()` 仍用于工程原有的 cross-page/vBerti 统计，不是 STLB-local useless 结算函数。

## 16. JSON、默认语义和模块发现

### 16.1 为什么 JSON 写完整路径

`prefetcher_stlb/` 是和原有 `prefetcher/` 并列的新模块根。[`Makefile`](/home/zcq/git_prj/ChampSim/Makefile:15) 将它加入搜索路径，但 JSON 仍使用 ChampSim 原有的 `prefetcher` key：

```json
"prefetcher": "prefetcher_stlb/sp"
```

这种写法是有效且当前推荐的，可以明确区分原有 data prefetcher 与 STLB-specific 目录。当前没有另造一个 `stlb_prefetcher` JSON key。

### 16.2 启用配置

最小关键配置是：

```json
{
  "STLB": {
    "prefetcher": "prefetcher_stlb/sp",
    "prefetch_activate": "LOAD",
    "prefetch_as_load": false,
    "pq_size": 16,
    "mshr_size": 16
  },

    "rq_size": 16,
    "pq_size": 16,
    "max_read": 2,
    "max_write": 2
  }
}
```

`sp` 可替换为 `dp/asp/stp/h2p/masp/stlb_stride`。完整可编译示例见 [`stlb-prefetchers.json`](/home/zcq/git_prj/ChampSim/test/config/compile-only/stlb-prefetchers.json:1)。

只配 STLB `pq_size=16`、不配 PTW `pq_size`，候选项可进 STLB internal PQ，但 STLB miss 不能被 PTW PQ 接收，将在 STLB pipeline 重试。因此启用 STLB-local predictor 时应显式配两级 PQ。

### 16.3 旧 JSON 的默认语义

[`default_stlb`](/home/zcq/git_prj/ChampSim/inc/defaults.hpp:112) 仍然是：

```text
prefetcher       = no
STLB pq_size     = 0
prefetch_as_load = false
```

PTW `pq_size` 如果 JSON 未写，默认仍为 0，见 [`ptw_queue_defaults()`](/home/zcq/git_prj/ChampSim/config/instantiation_file.py:258)。

因此旧 JSON：

- 不写 `STLB.prefetcher` -> 仍选择原有 `prefetcher/no`；
- 不需要新建 `prefetcher_stlb/no`；
- 不写 STLB/PTW PQ -> 容量仍为 0；
- 不会自动产生 STLB-local request。

兼容性承诺的范围是“原本就不在 STLB 上启用通用 data prefetcher 的旧配置”。如果某个历史 JSON 刻意把普通 data-prefetcher 模块挂在 STLB 上，现在 STLB lookup/fill 不再调它的普通 operate/fill，这不在 no-mode 语义不变承诺内。

## 17. 与 vBerti Permit-PGC 共存

两条链路可以同时开启：

| 属性 | vBerti Permit-PGC translation | STLB-local prefetcher |
|---|---|---|
| 预测器位置 | L1D data prefetcher | STLB module |
| 发射 API | 原有 `prefetch_line()`/跨页 translation 链路 | `prefetch_translation()` |
| origin | `L1D_PREFETCH_*` | `STLB_PREFETCH` |
| 到 PTW 的请求类型 | demand-like `LOAD` | `PREFETCH` |
| PTW queue | RQ | PQ |
| 统计 | 原有 vBerti/cross-page 统计 | 独立 `stlb_prefetch_*` 统计 |
| 是否训练 STLB predictor | 否 | 自己的 request 也不递归训练 |

语义上它们是可共存的，但性能上不是完全互不影响：它们会共享 STLB tag bandwidth/MSHR、PTW `MAX_READ/MAX_FILL` 和下层 cache/DRAM 资源。这种资源竞争是真实模型的一部分，不是统计混淆。

## 18. 当前 predictor 目录的解耦结构

[`prefetcher_stlb/`](/home/zcq/git_prj/ChampSim/prefetcher_stlb/README.md:1) 中每个 predictor 的状态、候选生成、replacement/table 和私有统计都放在自己目录下：

```text
prefetcher_stlb/sp/{sp.h,sp.cc}
prefetcher_stlb/dp/{dp.h,dp.cc}
prefetcher_stlb/asp/{asp.h,asp.cc}
prefetcher_stlb/stp/{stp.h,stp.cc}
prefetcher_stlb/h2p/{h2p.h,h2p.cc}
prefetcher_stlb/masp/{masp.h,masp.cc}
prefetcher_stlb/stlb_stride/{stlb_stride.h,stlb_stride.cc}
```

没有让多个算法依赖一个 `stlb_prefetcher_common.h`。模块之间共享的只有 ChampSim infra 公开的：

- `champsim::modules::stlb_prefetcher`；
- operate/fill context；
- `prefetch_translation()`。

这和原有 data prefetcher 的模块组织方式一致：模块只依赖 infra，算法私有代码不跨目录共享。

## 19. 当前明确的限制与后续扩展点

| 边界 | 当前状态 | 需要扩展时的做法 |
|---|---|---|
| 训练流 | Infra 可通知 hit/miss，六种基础算法只用 accepted demand miss | access-trained 算法可以删除 predictor 内的 `if (hit) return` |
| CPU/ASID predictor state | context 有信号，六种算法未做 per-core/per-ASID partition | multicore/multi-ASID 实验前按 `(cpu, asid)` 分区或 context switch flush |
| 延迟发射 | `prefetch_translation()` 使用隐式最近 CPU/ASID | 将 CPU/ASID 加入发射签名或 pending context |
| Metadata | 32 bit | 大状态放 predictor 内部，metadata 仅存 ID；或扩宽端到端字段 |
| 拒绝原因 | `bool` 只表示 STLB internal PQ 接收结果 | 只有算法/统计真正需要时再增加 result enum |
| PTW MSHR capacity | 有配置字段，当前 operate 未使用它限流 | 在 PTW 开始 walk/插入 MSHR 前强制容量 |
| Operate usefulness signal | `context.useful_prefetch` 仍来自通用 `way.prefetch` | 新增显式 `stlb_prefetch_useful_hit` |
| Fill ownership signal | `fill_context.prefetch` 是 MSHR effective type，不是 ownership | 新增 `stlb_prefetch_owned`/`stlb_prefetch_used` |
| ASID 功能正确性 | 当前验证范围是单核、单地址空间 trace | 多 ASID 前还要审核 STLB 本身的 tag/MSHR matching 是否纳入 ASID |

这些是当前实现的范围边界，不影响已实现的单核、同步 miss-stream predictor 实验，但在新算法依赖更强 context 或扩展到多地址空间前必须重新检查。

### 20.2 阅读代码时可使用的不变式

1. `issued <= requested`。
2. `hit + miss == lookups`。
3. `late <= useful`。
4. 同一 prefetch-owned MSHR 只能在首个 demand 到来时增加一次 useful/late。
5. 后来 merge 到 demand/vBerti MSHR 的 STLB-local prefetch 不拥有 fill。
6. STLB `internal_PQ` 满才直接体现为 `requested-issued` drop；下游暂时拒绝使用 retry/backpressure。
7. PTW 在同一周期先用 RQ 消耗 `MAX_READ`，再考虑 PQ。
8. `useful + useless == fill` 不是不变式，因为 useless 只在真实 eviction 时记录，且 phase 结束存在右删失。
9. 不配置 STLB predictor 时不应出现 STLB-local requested/issued，也不应新增 PTW PQ 流量。
10. STLB-local provenance 只用于归因，不改变 tag/channel 匹配与 replacement 结果。

## 22. 可配置 STLB-local prefetch buffer 的 infra 改动

这一部分 PB 位于 ChampSim infra，而不位于 `prefetcher_stlb/` 算法目录。这样同一个
算法无需维护 `fill`/`pb` 两份实现，JSON 可以独立切换 destination。

### 22.1 Builder 与 JSON

[`cache_builder.h`](/home/zcq/git_prj/ChampSim/inc/cache_builder.h) 增加：

```text
stlb_prefetch_destination::{STLB, PREFETCH_BUFFER}
set/reset_stlb_prefetch_destination_buffer()
stlb_prefetch_buffer_size()
stlb_prefetch_buffer_latency()
```

[`instantiation_file.py`](/home/zcq/git_prj/ChampSim/config/instantiation_file.py) 将以下 JSON
字段生成到 builder chain：

```text
stlb_prefetch_destination
stlb_prefetch_buffer_size
stlb_prefetch_buffer_latency
```

缺省 destination 是 `STLB`、size 是 0、latency 是 2；destination 不是 `PB` 时 size
和 latency 都不参与执行。生成器拒绝非 STLB 配置、非法 destination，以及 `PB +
size=0`，从配置阶段阻止静默误用。

### 22.2 Fill 重定向

[`CACHE::handle_fill()`](/home/zcq/git_prj/ChampSim/src/cache.cc) 在任何 victim selection
之前检查 `should_redirect_stlb_prefetch_buffer_fill()`。判定同时要求：

```text
cache is STLB
destination == PB
origin == STLB_PREFETCH
stlb_prefetch_tracked == true
tlb_ptw_real_demand_waiting == false
```

最后一个条件是 late 边界：若 demand 已经 merge 到预取 MSHR，不能再把 translation
藏到 PB 中，否则等待中的 real demand 无法被正常唤醒；该 fill 必须走原 STLB fill。

PB entry 包含 physical translation data、virtual address、metadata、CPU、两字节 ASID、
generation 和 ROI tracking bit。generation 保证一个已开始的 lookup 不会误删 lookup
期间插入的同 VPN 新版本。

### 22.3 Demand lookup

原 `do_handle_miss` 在调用 `handle_miss()` 前仅增加一个缺省关闭的分支：

```text
start_stlb_prefetch_buffer_lookup(pkt)
```

不启用 PB 时该函数立即返回 false，随后逐语句进入原 `handle_miss()`。启用时只接受
`DEMAND_DATA` 的 STLB tag miss，将 packet 和命中快照放进 timed lookup queue。

每次 `CACHE::operate()` 在普通 fill 之后调用
`complete_stlb_prefetch_buffer_lookups()`：

- hit：消费匹配 generation，按 demand provenance 填 STLB并返回上级；
- miss：调用原 `handle_miss()`，因此 MSHR merge、MSHR full、PTW RQ full、统计和
  instrumentation 都复用旧路径；
- miss 被 backpressure 拒绝：entry 保留到下一周期重试，predictor 尚未收到回调；
- miss 被接受：只回调一次 STLB predictor。

这实现了论文语义中的 `STLB lookup -> 2-cycle PB lookup -> demand PTW`，而不是把 PB
与 STLB 并行查找，也不是在 PTW 完成后额外增加两周期。

### 22.4 与算法接口的连接

[`stlb_prefetcher_context`](/home/zcq/git_prj/ChampSim/inc/modules.h) 只增加一个明确的：

```cpp
bool prefetch_buffer_hit;
```

direct-fill 模式继续使用 `hit && useful_prefetch` 表示及时预取；PB 模式则使用
`prefetch_buffer_hit` 表示论文 PQ hit。算法仍然只能通过 `prefetch_translation()`
发射，不知道也不控制 destination。

### 22.5 统计隔离和旧路径保护

PB-specific raw counter 放在 `cache_stats`，原 `stlb_prefetch_*` 质量计数仍作为统一的
算法质量统计。plain/JSON printer 只有在 PB raw counter 非零时才增加 PB 段落，因此
旧 JSON 的输出格式不变。

既有 vBerti CP-PB 的 map、fill 判定、计数器和 CLI 开关没有重构。若 CP-PB 与新 PB
同时包含一个 demand VPN，先让既有 CP-PB 按原零延迟路径命中；只有 CP-PB 不命中才
开始新 PB 的串行 lookup。
